One core, multiple models

Nexum is built around a single native engine. Graph traversal, relational rows, and typed object persistence all ride on the same storage, index, and recovery path, which keeps the system coherent under load and easier to reason about.

Browse the Nexum repository on GitHub for the current source, build scripts, and implementation details.

What to read first

Start with Getting Started for build and integration steps, continue with Architecture for layering, then use Unified Engine Model and the reference pages for the object, relational, concurrency, and query surfaces.

Jump straight into the code through the core storage package or the server module.

If you are thinking about server-side object access, review the Object Query API Design proposal.

Operating principles

  • Keep the engine record-based and facade-agnostic.
  • Keep schema and persistence semantics explicit.
  • Let storage and indexing do the heavy work beneath stable APIs.
  • Favor disciplined implementation over hidden framework behavior.
